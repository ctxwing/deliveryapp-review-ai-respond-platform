### 2.5 [핵심] 아키텍처 조립 전략 및 최종 권고

#### 🔵 Gemini 조립 전략

**조립 전략:** Cookiecutter Django를 메인 뼈대(Skeleton)로 삼아 PostgreSQL 기반의 멀티테넌시 구조를 확립. Playwright 기반의 수집 Unit과 **PydanticAI** 기반의 AI Unit을 결합.

**구체적 시나리오:**
1. **Skeleton (50%):** Django + Django-Tenants를 활용하여 본사/매장별 데이터 격리 및 권한 관리 구현
2. **Scraper Block (20%):** Celery Beat를 이용하여 주기적으로 배달앱 스크래핑 수행
3. **AI Block (15%):** **PydanticAI**로 리뷰 감성 분석 후 브랜드별 페르소나를 주입하여 답글 생성 (타입 안전한 구조화된 출력)
4. **Admin Block (15%):** Refine 또는 Ant Design Pro를 사용하여 승인 대기열 및 통계 대시보드 구축

**커스텀 비중:** 오픈소스 활용 70%, 자체 비즈니스 로직 30%

#### 🟢 OpenAI 조립 전략

**구조 요약:**
```
[Review Source]
   ↓
[Collector (Apify/Crawler)]
   ↓
[Django API]
   ↓
[AI Engine (PydanticAI)]
   ↓
[Policy Engine]
   ↓
[DB + Dashboard]
```

**수행해야 할 이유 (Pros):**
1. **SaaS 확장성:** 프랜차이즈 시장의 고질적 페인포인트를 해결하는 모델로, 솔루션화 시 지속적 수익 창출 가능
2. **기술 자산 확보:** 배달 플랫폼 스크래핑 및 LLM 답글 생성 로직은 타 도메인(e-커머스 등)에 즉시 재활용 가능
3. **시장 수요 폭발:** AI 자동 응대 솔루션은 현재 가장 활발하게 도입되는 분야로 포트폴리오 가치가 매우 높음

**하지 말아야 할 이유 (Cons):**
1. **스크래핑 리스크:** 배달 플랫폼의 빈번한 UI 변경 및 차단 정책으로 인해 유지보수 공수가 예상보다 클 수 있음
2. **예산 대비 과업 과다:** 앱 개발(Android/iOS)까지 포함된 금액으로는 인건비 확보가 어려움 (웹 우선 개발로 협의 필수)
3. **답글 품질 책임:** AI가 생성한 부적절한 답글로 인한 브랜드 이미지 훼손 시 법적/도의적 책임 소재 분쟁 가능성 존재

### 🟢 OpenAI 결론 근거

**해야하는 이유:**
1. SaaS 확장 → 반복 수익 가능
2. AI + 크롤링 → 진입장벽 높음
3. 프랜차이즈 시장 → 지속 수요

**리스크:**
1. 배달앱 API 없음 → 크롤링 리스크
2. AI 품질 → 브랜드 리스크
3. 정책 복잡도 → 유지보수 비용 증가
