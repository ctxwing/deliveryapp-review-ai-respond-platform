# 프로젝트 헌장 (Project Constitution)

**프로젝트명**: AI기반 리뷰 통합 관리 플랫폼 구축
**코드명**: delivery-ai-responder-platform
**버전**: v1.1.0
**비준일**: 2026-04-01
**최종 수정일**: 2026-04-01

---

## 프로젝트 개요

배달앱 및 리뷰 플랫폼에 등록된 고객 리뷰를 수집·통합하고, AI가 리뷰 내용을 분석하여 적절한 답글을 생성하며, 설정된 정책에 따라 자동 등록 또는 관리자 승인 후 등록할 수 있도록 지원하는 리뷰 운영 자동화 서비스입니다.

- **예산**: 20,000,000원
- **기간**: 120일 (약 17주)
- **기술 스택**: Python/Django, React/TS, PostgreSQL, AWS
- **지원 디바이스**: 웹 기반 관리자 서비스 (향후 모바일 확장 고려)
- **발주 형태**: 상세 기획, 디자인, 개발, 인프라 구성 전체

---

## 원칙

### 원칙 1: 한글 작성 및 문서화

모든 문서, 코드 주석, 커밋 메시지, PR 설명은 **한글**로 작성합니다.
작성자는 `ctxwing@gmail.com`으로 통일하며, AI가 자동 생성하는 문서에
"작성자: AI" 등의 서명을 추가하지 않습니다.

- 모든 커밋 메시지 MUST 한글로 작성
- 코드 내 인라인 주석 MUST 한글로 작성
- 외부 라이브러리 원문 인용 시에만 영문 허용

**근거**: 국내 프랜차이즈 고객사 대상 프로젝트로, 인수인계 시
한글 문서화가 필수적이며 유지보수 편의성을 보장합니다.

### 원칙 2: 시간대 표준 (KST)

모든 시간 관련 데이터 및 로그는 **KST (Asia/Seoul, UTC+9)**를 기준으로 합니다.
Docker 컨테이너 사용 시 `TZ=Asia/Seoul` 환경변수를 MUST 설정합니다.

- 리뷰 작성 일시, 답글 생성 일시, 로그 타임스탬프 MUST KST 표기
- ISO 8601 형식 사용 시 `+09:00` 오프셋 MUST 명시

**근거**: 고객사와 서버가 모두 한국 시간대를 사용하며,
리뷰 수집 및 답글 등록 시간 혼동을 방지합니다.

### 원칙 3: 단일 브랜치 운영 (main)

로컬과 리모트 모두 오직 **`main`** 브랜치 하나만 운영합니다.
자동화 도구나 워크플로우를 사용할 때에도 새로운 기능 브랜치를
절대 생성하지 않고 항상 `main` 브랜치에서 작업합니다.

- feature, develop, release 등 별도 브랜치 MUST NOT 생성
- 모든 커밋은 `main`에 직접 푸시
- rebase 기반 히스토리 관리 권장

**근거**: 소규모 개발팀 특성상 브랜치 전략의 오버헤드를 제거하고,
빠른 이터레이션을 가능하게 합니다.

### 원칙 4: Django 기반 백엔드 표준

백엔드는 **Python/Django**를 표준 프레임워크로 사용합니다.
고객 요구사항에 명시된 기술 스택을 준수하여 타 언어/프레임워크로의
교체를 금지합니다.

- Django REST Framework (DRF) MUST 사용
- PostgreSQL MUST 사용 (데이터베이스)
- Celery + Redis MUST 사용 (비동기 작업 큐)

**근거**: 고객 요구사항에 "Python / Django"가 명시되어 있으며,
Django 생태계의 안정성과 멀티테넌시 지원(Django-Tenants)이 검증되었습니다.

### 원칙 5: LLM 기반 AI 엔진 (PydanticAI)

리뷰 답글 생성은 **외부 LLM API**를 활용합니다.
**PydanticAI**를 통한 타입 안전한 오케스트레이션으로
프롬프트 관리와 응답 검증을 체계화합니다.

- OpenAI/Claude API MUST 활용 (상용 LLM)
- **PydanticAI MUST 사용** (타입 안전한 구조화된 출력)
- Pydantic BaseModel으로 응답 스키마 정의 MUST
- 브랜드별 톤앤매너 MUST 반영 (system_prompt)
- 금지어, 민감 표현 MUST 필터링

**근거**: 리뷰 답글은 구조화된 출력이 핵심이며, PydanticAI의
타입 안전성이 런타임 에러를 방지합니다. LangChain 대비 코드가
간결하고 유지보수에 유리합니다.

### 원칙 6: 멀티테넌시 SaaS 구조

본사-가맹점 계정 체계를 **멀티테넌트 구조**로 설계합니다.
데이터 격리와 권한 관리를 DB 수준에서 보장합니다.

- Django-Tenants MUST 사용 (테넌트 격리)
- Brand → Store 1:N 관계 MUST 유지
- Row-level 접근 제어 MUST 구현
- 향후 SaaS 과금 모델 고려한 구조

**근거**: 프랜차이즈 본사가 여러 가맹점을 관리하는 구조이며,
향후 SaaS 서비스로의 확장성이 핵심 요구사항입니다.

### 원칙 7: 웹 우선 개발

**웹 기반 관리자 서비스**를 우선 개발합니다.
모바일 앱(Android/iOS) 개발은 범위에서 제외하거나 향후 확장으로
고려합니다.

- PC Web 관리자 페이지 MUST 최우선 개발
- 반응형 UI로 태블릿 지원 권장
- 모바일 앱은 Phase 2 이후 고려

**근거**: 예산(2,000만원) 대비 웹+앱 동시 개발은 인건비 확보가 어려우며,
설계분석에서도 "웹 우선 개발로 협의 필수"로 제안되었습니다.

### 원칙 8: 크롤링 안정성 최우선

배달앱 리뷰 수집은 **가장 큰 리스크**로 인식하고 안정성을 최우선으로
고려합니다.

- Playwright 또는 Apify MUST 사용 (Headless Browser)
- Proxy / Rotating IP MUST 활용 (차단 우회)
- 수집 실패 시 알림 및 재시도 로직 MUST 구현
- DOM 변경에 대응하는 유연한 파서 구조

**근거**: 설계분석에서 "이 프로젝트는 AI보다 크롤링이 더 어렵고,
성공 여부는 수집 안정성에서 결정됨"이 핵심 결론입니다.

### 원칙 9: AI 답글 품질 및 책임 관리

AI가 생성한 답글로 인한 브랜드 이미지 훼손 리스크를 관리합니다.

- 저별점(1~3점) 리뷰는 MUST 수동 검토 (자동 등록 금지)
- 민감 키워드(환불, 위생, 이물질, 신고, 불친절 등) 포함 시 MUST 수동 검토
- 관리자 수정 기능 MUST 제공
- 재생성 기능 MUST 제공
- AI 생성 로그(prompt/response) MUST 저장 (책임 소재 명확화)

**근거**: 설계분석에서 "AI가 생성한 부적절한 답글로 인한 브랜드 이미지
훼손 시 법적/도의적 책임 소재 분쟁 가능성"이 주요 리스크로 식별되었습니다.

## 거버넌스

### 수정 절차

1. 원칙 수정이 필요한 경우 `openspec` 프로포절을 작성합니다.
2. 프로포절 승인 후 본 헌장 문서를 갱신하고 버전을 올립니다.
3. 의존 템플릿(plan-template, spec-template, tasks-template)의
   Constitution Check / Constraints 섹션을 동기화합니다.

### 버전 관리 정책

- **MAJOR**: 원칙의 삭제 또는 의미가 역전되는 재정의
- **MINOR**: 새 원칙 추가 또는 기존 원칙의 실질적 범위 확장
- **PATCH**: 문구 명확화, 오타 수정, 비의미적 개선

### 준수 검토

- 매 spec/plan 작성 시 "Constitution Check" 섹션에서 전 원칙 준수 여부를 확인
- tasks.md의 T000 태스크에서 헌장 준수 초기 점검 수행
- 위반 시 Complexity Tracking 표에 위반 사유와 대안 거부 이유를 기록
