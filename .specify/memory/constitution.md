# 프로젝트 헌장 (Project Constitution)

**프로젝트명**: 3D AR 뷰어 상세페이지 (3d-ar-viewer-detailpage)
**코드명**: 3d-ar-viewer-detailpage
**버전**: v1.0.0
**비준일**: 2026-03-31
**최종 수정일**: 2026-03-31

---

## 프로젝트 개요

자사몰(카고컨테이너) 내 주요 캠핑 기어 4종(파이어핏맥스, 실드박스, 사이드 플립 테이블, 스퀘어 X)에 대해
3D/AR 디지털 트윈을 구축하고, 각 제품 상세페이지에 웹 기반 3D 뷰어 및 AR 미리보기 기능을 구현하여
사용자의 온라인 쇼핑 경험을 강화하는 Turnkey 프로젝트이다.

- **예산**: 3,000,000원
- **기간**: 2026-03-29 ~ 2026-06-30 (약 13주)
- **유지보수**: 런칭 후 3개월 무상
- **발주 형태**: Turnkey (설계 → 개발 → 테스트 → 유지보수 전체 책임)

---

## 원칙

### 원칙 1: 한글 작성 및 작성자 표기

모든 문서, 코드 주석, 커밋 메시지, PR 설명은 **한글**로 작성한다.
작성자는 `ctxwing@gmail.com`으로 통일하며, AI가 자동 생성하는 문서에
"작성자: AI" 등의 서명을 추가하지 않는다.

- 모든 커밋 메시지 MUST 한글로 작성
- 코드 내 인라인 주석 MUST 한글로 작성
- 외부 라이브러리 원문 인용 시에만 영문 허용

**근거**: 국내 고객사 대상 Turnkey 프로젝트로, 인수인계 시
한글 문서화가 필수적이며 유지보수 편의성을 보장한다.

### 원칙 2: 시간대 표준 (KST)

모든 시간 관련 데이터 및 로그는 **KST (Asia/Seoul, UTC+9)**를 기준으로 한다.
Docker 컨테이너 사용 시 `TZ=Asia/Seoul` 환경변수를 MUST 설정한다.

- 파일 생성 일시, 로그 타임스탬프 MUST KST 표기
- ISO 8601 형식 사용 시 `+09:00` 오프셋 MUST 명시

**근거**: 개발 서버(ctx-tower)와 고객사 모두 한국 시간대를 사용하며,
시간 혼동을 방지한다.

### 원칙 3: 단일 브랜치 운영 (main)

로컬과 리모트 모두 오직 **`main`** 브랜치 하나만 운영한다.
자동화 도구나 워크플로우를 사용할 때에도 새로운 기능 브랜치를
절대 생성하지 않고 항상 `main` 브랜치에서 작업한다.

- feature, develop, release 등 별도 브랜치 MUST NOT 생성
- 모든 커밋은 `main`에 직접 푸시
- rebase 기반 히스토리 관리 권장

**근거**: 소규모 1인 개발 프로젝트 특성상 브랜치 전략의 오버헤드를
제거하고, 빠른 이터레이션을 가능하게 한다.

### 원칙 4: 오픈소스 우선 기술 스택

핵심 기술은 오픈소스 기반을 우선 채택하여 라이선스 비용을 절감하고,
예산 제약(300만원) 내에서 최대 품질을 달성한다.

| 영역 | 필수 기술 | 라이선스 |
|------|-----------|----------|
| 3D/AR 뷰어 | Google `<model-viewer>` | Apache-2.0 |
| 3D 모델링 | Blender | GPL-2.0 |
| 메시 압축 | Draco 3D | Apache-2.0 |
| 텍스처링 | Substance Painter 또는 오픈소스 대안 | 상용/선택 |
| 모델 최적화 | glTF-Pipeline | Apache-2.0 |
| 커스텀 UI | Three.js (선택적 확장) | MIT |

- 상용 SaaS(Sketchfab Enterprise, Vectary 등) MUST NOT 런타임 의존으로 도입
- Vuforia MUST NOT 사용 (웹 기반 AR에 부적합, `<model-viewer>` AR로 대체)
- 추가 의존성 도입 시 Apache-2.0, MIT, BSD 등 허용 라이선스 MUST 확인

**근거**: 설계분석에서 `<model-viewer>`가 오늘의집 AR과 동일 엔진이며,
iOS(USDZ)/Android(GLB) AR을 내장 지원하여 Vuforia 없이 요구사항을
충족할 수 있음이 확인되었다.

### 원칙 5: 모바일 퍼포먼스 우선 최적화

3D 에셋과 웹 뷰어는 **모바일 환경 로딩 속도**를 최우선으로 최적화한다.
상세페이지 이탈을 방지하기 위해 초기 로딩과 인터랙션 반응성에 집중한다.

- 제품당 GLB 파일 크기 MUST ≤ 5MB (Draco 압축 적용 후)
- 3D 뷰어 초기 로딩 시간 MUST ≤ 3초 (4G LTE 기준)
- Low-Poly 모델링 후 노멀맵 베이킹으로 디테일 보존
- 텍스처 해상도: 모바일용 1K (1024×1024), PC용 2K (2048×2048) 분리 제공
- Lazy Loading 및 프로그레시브 로딩 전략 MUST 적용

**근거**: 설계분석에서 "파일 용량 문제로 인한 모바일 로딩 속도 저하 →
고객 불만 발생 리스크"가 주요 Cons로 식별되었다.

### 원칙 6: 크로스 플랫폼 AR 호환성

AR 기능은 별도 앱 설치 없이 **모바일 웹 브라우저**에서 작동해야 한다.
iOS와 Android 양 플랫폼을 동시에 지원하되, 비지원 환경에서는
3D 뷰어로 우아하게 폴백한다.

- iOS: USDZ 기반 AR Quick Look MUST 지원
- Android: GLB 기반 Scene Viewer MUST 지원
- AR 비지원 브라우저: 3D 뷰어만 표시 (AR 버튼 숨김)
- QA 대상 브라우저: Safari(iOS), Chrome(Android), 삼성 인터넷, 인앱 브라우저
- 기존 쇼핑몰에 `<iframe>` 또는 Web Component 방식으로 임베딩하여
  어드민 수정을 최소화

**근거**: 고객 요구사항에서 "오늘의집, 리모아 사례와 유사한 서비스"를
명시하였으며, 이들 모두 웹 기반 AR을 사용한다.

### 원칙 7: 완료 내역 문서화

주요 마일스톤 완료 시 `3_prj_docs/완료내역/` 디렉토리에
완료 보고서를 작성하여 고객 인수인계 및 유지보수에 대비한다.

- 마일스톤별 완료 보고서 MUST 작성
- 보고서에 스크린샷/동영상 첨부 권장
- QA 테스트 결과(기기별 AR 작동 여부) MUST 포함

**근거**: Turnkey 방식 프로젝트로 설계부터 유지보수까지 전체 책임을 지며,
체계적인 문서화가 분쟁 방지 및 3개월 무상 유지보수 범위 명확화에 필수적이다.

## 거버넌스

### 수정 절차

1. 원칙 수정이 필요한 경우 `openspec` 프로포절을 작성한다.
2. 프로포절 승인 후 본 헌장 문서를 갱신하고 버전을 올린다.
3. 의존 템플릿(plan-template, spec-template, tasks-template)의
   Constitution Check / Constraints 섹션을 동기화한다.

### 버전 관리 정책

- **MAJOR**: 원칙의 삭제 또는 의미가 역전되는 재정의
- **MINOR**: 새 원칙 추가 또는 기존 원칙의 실질적 범위 확장
- **PATCH**: 문구 명확화, 오타 수정, 비의미적 개선

### 준수 검토

- 매 spec/plan 작성 시 "Constitution Check" 섹션에서 전 원칙 준수 여부를 확인
- tasks.md의 T000 태스크에서 헌장 준수 초기 점검 수행
- 위반 시 Complexity Tracking 표에 위반 사유와 대안 거부 이유를 기록
